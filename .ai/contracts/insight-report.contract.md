# Insight Report UI Contract

## Status

Draft for TASK-002 planning. This is a UI-facing contract document, not runtime TypeScript source.

## Purpose

Define the frontend-facing Product Signal Report shapes and API integration assumptions for the future NoiseCut Web App. The Founder lens is the v0 priority. API generation, persistence, telemetry, and public backend contracts remain owned by `C:\Dev\NoiseCut\noise-cut-api`.

## Compatibility Baseline

- Reuse existing API capabilities where possible: `GET /ready`, `GET /api/meta`, `POST /api/analysis-jobs`, `GET /api/analysis-jobs/{jobId}`, `POST /api/v1/product-insights/analyze`, `POST /api/v1/product-insights/verdict`, and `POST /api/feedback`.
- Do not break deprecated compatibility wrappers: `POST /api/v1/analyze`, `POST /api/analyze`, `POST /api/v1/analyze/verdict`, and `POST /api/analyze/verdict`.
- Preserve extension behavior: readiness check, metadata compatibility check, background job creation, job polling, saved `jobId`/`contentHash`, and feedback submission.
- Treat any report-specific API behavior as future, additive, and API-repo-owned.

## UI Types

These shapes describe what the UI should eventually consume and produce. Final runtime placement depends on the future app scaffold.

```ts
export type InsightPlatform = "reddit" | "youtube"

export type InsightLens = "founder" | "creator" | "marketing"

export type InsightSourceItem = {
  id?: string
  text: string
  author?: string
  score?: number
  depth?: number
  createdAt?: string
  metadata?: Record<string, string>
}

export type InsightReportRequest = {
  platform: InsightPlatform
  url?: string
  lens: InsightLens
  researchContext?: string
  title?: string
  items?: InsightSourceItem[]
  comments?: InsightSourceItem[]
  metadata?: Record<string, string>
  correlationId?: string
}

export type InsightReportItem = {
  title?: string
  text: string
  explanation: string
  evidenceCount?: number
  confidence?: string
  sourceReferences?: string[]
}

export type InsightReportResponse = {
  summary: string
  painPoints: InsightReportItem[]
  featureRequests: InsightReportItem[]
  complaints: InsightReportItem[]
  objections: InsightReportItem[]
  competitorMentions: InsightReportItem[]
  demandSignals: InsightReportItem[]
  userLanguage: InsightReportItem[]
  contentIdeas: InsightReportItem[]
  productOpportunities: InsightReportItem[]
  limitations: InsightReportItem[]
  metadata: Record<string, string>
}
```

## Lens Rules

- `founder` is the only committed v0 lens.
- `creator` and `marketing` are included to avoid immediate type churn, but UI behavior should hide or disable them until a later scope enables them.
- Future API metadata should be checked before showing non-Founder lenses.

## Request Semantics

- `platform` is required and must be `reddit` or `youtube`.
- `url` is optional to support future extension handoff or direct extracted-content payloads, but URL-first web app flows should provide it.
- `researchContext` is optional user-provided context and must be treated as sensitive.
- `items` and `comments` are optional extracted evidence arrays for future extension handoff or UI/API extraction flows.
- `metadata` is for privacy-safe client/source metadata only.
- `correlationId` may be generated by the UI or preserved from a caller/handoff flow.

## Response Semantics

- `summary` is the executive summary for the Founder report.
- Section arrays hold structured insight items that can be rendered in the UI and converted to Markdown.
- `sourceReferences` should use lightweight references or labels, not raw content dumps.
- `metadata` should carry non-content report metadata such as platform, lens, job/report identifiers, generated timestamp, model/report version when provided by the API, and capability flags.

## UI API Client Responsibilities

- Read API base URL and timeout from browser-safe configuration.
- Check `GET /ready` before report generation.
- Check `GET /api/meta` before assuming platform, analysis type, or report support.
- Prefer the existing background-job flow if the API can return the report through job status/result.
- Preserve or send `correlationId` where supported.
- Normalize API errors into UI-safe messages.
- Return structured `InsightReportResponse` data or a documented wrapper if the API uses `jobId`, `contentHash`, and status around the report.
- Keep feedback compatible with `POST /api/feedback`.

## Environment And Config Assumptions

- Future browser-safe API base URL config: `NOISECUT_API_BASE_URL` or the chosen framework's public equivalent.
- Future timeout config: `NOISECUT_API_TIMEOUT_MS` or equivalent.
- No browser-exposed secrets.
- No auth, billing, deployment, or persistence config in this contract.

## Privacy Rules

- Do not log raw page, post, comment, transcript, prompt, provider response, `researchContext`, `items`, or `comments` in the UI.
- Do not store raw extracted content in browser storage by default.
- Keep feedback lightweight and reference-based unless a later privacy-reviewed task expands it.
- API storage and retention behavior must be defined in the API repo.

## Observability Rules

- Privacy-safe dimensions may include platform, lens, UI surface, request status, job status, outcome, error code, and latency bucket.
- Propagate `correlationId` when supported.
- Avoid raw URLs in telemetry unless a later telemetry policy explicitly allows them.
- Align with API telemetry names where possible.

## Open Questions

- Whether the API exposes the full report through the existing job model, a new report analysis type, or another additive endpoint.
- Whether Markdown export is generated by the API or assembled by the UI.
- Whether URL-only report generation requires API-side extraction or UI/extension-supplied evidence.
- Whether the eventual runtime type should live in a shared UI package, API client module, or generated contract file after app scaffolding.
